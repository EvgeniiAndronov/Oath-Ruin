# ADR-001: Монолитная архитектура (Monorepo)

**Статус:** Принято  
**Дата:** 2026-03-05  
**Автор:** Evgenii Andronov

---

## Контекст

Нужно выбрать структуру репозитория для игры Oath & Ruin:
- Один репозиторий для клиента и сервера (Monorepo)
- Или раздельные репозитории

Команда: 1 разработчик + дизайнеры (позже).
Цель: Обучение PM и архитектуре, быстрая разработка MVP.

---

## Решение

**Использовать Monorepo** — клиент и сервер в одном репозитории.

Структура:
```
oath-and-ruin/
├── client/          # Godot 4.x
├── server/          # Go 1.21+
├── docs/            # Документация
└── README.md
```

---

## Альтернативы

### 1. Раздельные репозитории
```
oath-and-ruin-client/
oath-and-ruin-server/
```

**Плюсы:**
- Чёткое разделение ответственности
- Независимый CI/CD
- Меньше размер каждого репо

**Минусы:**
- Сложнее синхронизировать версии API
- Два места для управления
- Отдельный versioning

### 2. Monorepo с workspace (Go workspaces)
```
go.work
├── client/  (если будет Go сервер в клиенте)
├── server/
└── libs/    (общие библиотеки)
```

**Плюсы:**
- Общие зависимости
- Удобный рефакторинг

**Минусы:**
- Сложнее настройка
- Не нужно для текущего масштаба

---

## Последствия

### Положительные
+ ✅ Всё в одном месте (проще для одного разработчика)
+ ✅ Версии клиента и сервера синхронизированы
+ ✅ Один CI/CD пайплайн
+ ✅ Легко рефактить API (видишь изменения вместе)
+ ✅ Проще документировать

### Отрицательные
- ❌ Репозиторий будет расти (но для инди это не проблема)
- ❌ Сложнее разделить команду потом (но это решаемо через git subtree split)
- ❌ Разные языки в одном месте (Godot + Go)

---

## Критерии для разделения в будущем

Разделим на два репозитория, если:
1. Придёт вторая команда на сервер (3+ человека)
2. Репозиторий станет >1GB
3. Понадобится разный доступ (client devs ≠ server devs)
4. Сервер будет использоваться для других проектов
